# Chapter 10 — UI and Client State Assumptions

This chapter defines binding client-facing UI behavior: UI state model, panel rules (HUD/chat), modal behavior, accessibility, and “what the player sees / what is clickable.”

**Scope boundary (binding):**
- Chapter 10 owns UI behavior only.
- Chapter 11 owns failure-state taxonomy and backend recovery contracts.
- Chapter 08 owns chat routing/eligibility/retention.
- Chapter 03 owns season-scoped action legality and activity enforcement.
- Chapter 15 owns reason-code catalogs and telemetry schema.

---

## 10.1 Global vs Seasonal Context

UI tracks two independent contexts:

1) **Viewing context**
- `Viewing: Global` iff `viewing.active_season_id == null`.
- `Viewing: Season <N>` iff `viewing.active_season_id == <N>`.

2) **Participation context**
- `Participation: None` iff `GlobalParticipationStatus == false` (defined in Chapter 15).
- `Participation: Season <A>` iff `GlobalParticipationStatus == true`, where `A == participation.joined_season_id`.

**GlobalParticipationStatus derivation (binding; owner = Chapter 15):**
See Chapter 15 §15.11.5 (Derived client fields).

**Display rule (binding):**
- The Season Selector surface (or equivalent always-reachable navigation control) MUST be able to show both contexts simultaneously when they differ:
  - `Viewing: Season <N>` and `Participation: Season <A>` where `A != N`.

This rule MUST NOT override the Chapter 09 status-chip rules for any header-level chip strip. (Chapter 09 controls header-level status chips.)

---

## 10.2 Real-Time Surfaces and Authority

- All economic truth is server-authoritative; client MUST NOT invent costs, rewards, legality, or ordering.
- Leaderboard ordering is server-authoritative; client MUST NOT reorder rows locally.
- Chat is wall-clock real-time (Chapter 08) and MUST NOT be delayed by season tick processing.

---

## 10.3 Required Always-Visible Primitives (Minimum)

- Global Stars balance is always visible in Global and Season viewing contexts.
- Seasonal Stars balance is visible in Season viewing context.
- Coins are visible only to **self** and only for the player’s **Participation** season (Chapter 04).
  - If `Participation: Season <A>` and `Viewing: Season <B>` where `A != B`, the UI MUST still show the player’s Coins for season `<A>` (not for `<B>`).

### 10.3.1 UI Magnitude Shorthand (T+)

For very large numeric values, the UI MAY render a shorthand magnitude display. This is **rendering-only**.

**Trigger.**
- If a numeric value has **≥ 13 digits** (i.e., `value >= 10^12`), the UI MAY render shorthand.

**Format.**
- Render as: `floor(value / 10^12) + "T+"` (no decimals).

**Scope.**
- This shorthand MAY be applied to:
  - Coins
  - Seasonal Stars
  - Global Stars
  - Sigil inventory tier counters (per-tier counts)

**Non-authoritative guarantee.**
- Shorthand displays MUST NOT replace or modify stored values.
- Shorthand displays MUST NOT participate in calculations, pricing, fees, ordering, or leaderboards.
- All ordering logic MUST use stored numeric values (never UI renderings).

---

## 10.4 Status Chips and Header Rules (Binding)

Any global header status chip strip MUST obey Chapter 09 §9.4.2 (max 2 chips: Participation mode + Constraint state).

**Participation mode chip (header-level):**
- MUST reflect **GlobalParticipationStatus** (Chapter 15), not the currently viewed season.

**Constraint state chip (header-level):**
- If disconnected, show `Disconnected` and do not also show `Idle`.
- If idle (and not disconnected), show `Idle (activity required)`.

This chapter MUST NOT introduce any additional header-level chip types that violate the “max 2” rule.

---

## 10.5 Modals and Freeze Behavior (UX Only)

### 10.5.1 Modal triggers (predicates)

- Disconnect prompt: transport disconnected (client connectivity state).
- Idle prompt: server flag `idle_modal_active == true` (do not infer).

### 10.5.2 Modal precedence

When a modal is visible, the header MUST render only:
- `State: Disconnected` or
- `State: Idle`

Countdown is not rendered while a modal is visible.

### 10.5.3 Modal interaction rules (binding)

Disconnect modal:
- Blocks all interaction behind modal.
- Buttons: `Retry`, `Logout`.

Idle modal (critical change for chat real-time):
- Blocks **season-scoped economy actions** behind the modal.
- **Does NOT block chat panels.**
  - Players MAY read and send Global chat and DMs while Idle modal is visible.
  - If the player is participating in a season, season chat remains available per Chapter 08 eligibility.
  - **Binding:** If `Viewing` season != `Participation` season, the Season chat tab MUST bind to the **Participation** season (not the Viewing season).
- Buttons: `I’m here` (issues `idle_modal_ack`), `Logout`.

**Idle does not clear via chat.**
- Chat never counts-for-activity (Chapter 08).
- Clearing idle requires `idle_modal_ack` committing (Chapter 03).

### 10.5.4 Allowed interactions while Idle modal is visible

Allowed:
- Chat panel: scroll, switch tabs (Global/DM/Season chat where eligible), send messages.
- `I’m here` (ack), `Logout`, accessibility settings.

Forbidden:
- Any season-scoped economy action controls (star purchase, vault purchase, boost purchase, trade, lock-in, join/re-enter, etc.).

### 10.5.5 Chat UI treatment while Idle

- Chat input MUST remain enabled.
- If a send is rejected by the server, client MUST show a toast using server `reason_code` and refresh snapshot.

---

## 10.6 Countdown and Urgency (Summary)

- Countdown source-of-truth is server field `remaining_seconds` (Chapter 07).
- Binding:
  - If `Viewing: Season <N>`: show remaining for viewing season.
  - If `Viewing: Global` and participating: show remaining for participating season.
  - Else: no countdown.
- Countdown may interpolate between server updates, but must freeze interpolation while a modal is visible.

---

## 10.7 Server-Flag Gating Requirement (Economy actions only)

Every **economy** actionable control (i.e., any control that triggers a Chapter 03 closed-set season action) MUST be backed by server-returned gating flags:

- `can_X` (boolean) where `X` is the action id (e.g., `can_trade_initiate`)
- `cannot_X_reason` (string `reason_code`; present iff `can_X == false`)

Client MAY be more restrictive due to local UI mode, but MUST NOT be less restrictive than server flags.

Chat/messaging controls (Chapter 08) are explicitly exempt:
- `chat_send` MAY be attempted and MAY be rejected without precomputed `can_chat_send` flags.


---

## 10.8 UI State Persistence Rules (Binding)

This section defines the binding minimum for what the client MAY persist locally and how it MUST recover.

### 10.8.1 Locally persisted UI preferences (allowed)

The client MAY persist the following UI-only fields locally (e.g., localStorage/device storage):

- last selected `viewing.active_season_id` (viewing selection)
- whether Chat panel is open (mobile toggle)
- last selected chat tab (`GLOBAL` / `DM` / `SEASON`)
- basic accessibility preferences (font scale, reduced motion, etc.)

These MUST be treated as **hints only**.

### 10.8.2 Recovery rule (authoritative)

On any app launch, reload, or reconnect, the client MUST:

1. Fetch the authoritative server snapshot (including participation state and gating flags).
2. Reconcile local hints against server truth:
   - If server indicates `Participation: Season <A>`, the UI MUST reflect participation in <A> regardless of any persisted values.
   - If persisted `viewing.active_season_id` points to a season that does not exist or is not returned by the server, the client MUST fall back to `Viewing: Global`.

The client MUST NOT assume it is participating based on local storage.

### 10.8.3 Multi-tab / multi-session behavior (binding)

If multiple sessions/tabs are open:

- Each tab MAY have its own `Viewing` selection.
- Participation is global per account and MUST remain consistent across tabs.
- Any server state change (e.g., join/exit/idle enforced) MUST be reflected after the next server snapshot refresh on each tab.

---

## 10.9 Mobile vs Desktop (Minimum)

- Mobile: only one of Chat or HUD is visible at a time.
- Desktop: Chat and HUD may both be visible.

Modal behavior applies equally on all mediums.
