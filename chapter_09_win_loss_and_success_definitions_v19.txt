# CHAPTER 09 — WIN, LOSS, AND SUCCESS DEFINITIONS

This chapter defines what “winning” means in Too Many Coins, what “losing” means, and how the game MUST frame outcomes.

This is a player-experience and retention chapter.
It binds official language rules and prohibits failure framing for player outcomes.

---

## 9.1 What Winning Means

Winning is defined in two contexts:

- **Season-winning (podium finish):** Finishing #1, #2, or #3 in a season’s final ordering at natural end.
- **Year-winning (global top rank):** Finishing #1 on the global leaderboard at yearly reset execution.

Both are valid “wins,” but neither implies all other outcomes are losses.

### 9.1.1 Season Winner

A season winner is:

- **Season Winner** = #1 at season finalization (natural end)
- **Season Runner-Up** = #2 at season finalization
- **Season Third Place** = #3 at season finalization

### 9.1.2 Year Winner

A year winner is:

- **Year Winner** = #1 on global leaderboard at yearly reset execution

Yearly badges are awarded to top placements as defined in Chapter 06.

### 9.1.3 “Current Leader” Labels

The UI may show “current leader” labels only as a snapshot-view label:

- **Season Current Leader** = the player who is rank **#1** on the **committed seasonal leaderboard snapshot being viewed** at the time of viewing.
- **Year Current Leader** = the player who is rank **#1** on the **committed global leaderboard snapshot being viewed** at the time of viewing.

Constraints:
- “Current leader” **MUST NOT** imply the player will win the season/year.
- “Current leader” **MUST NOT** imply any locked or guaranteed outcome.

Visibility:
- Current leader labels **MAY** be visible to **anyone**, including spectators, subject to multiplayer/social visibility rules (Chapter 08).

Slice/pagination constraint:
- If the UI is viewing a **partial leaderboard slice** that does not include the snapshot’s global **rank #1**, the UI **MAY** show a “current leader” label **only if** the server provides explicit authoritative leader fields for that committed snapshot: `leader_player_id` and `leader_identity_label` (Chapter 15).
- If no explicit leader identifier is provided, the UI **MUST NOT** show a “current leader” label unless the UI has loaded the snapshot’s global #1 row.

Empty/top-zero suppression:
- If the committed leaderboard snapshot contains no entries, the UI \*\*MUST NOT\*\* show any “current leader” label\.

- If the viewed leaderboard is the **yearly Global leaderboard** and the committed snapshot contains no entries, the UI MUST show the neutral header label: `No standings yet this year.`
- If the top value for the viewed snapshot is `0` (season: top `authoritative_seasonal_value`; year: top `GlobalStars`), the UI **MUST NOT** show any “current leader” label.

Post-season constraint:
- After a season has ended and final outcomes exist, UI **MUST** prefer end-only labels (e.g., **Season Winner**) over “Season Current Leader.”

## 9.2 What Losing Means

The game defines **no official “loss” state**.

### 9.2.1 No Formal Loss Outcome

The system **MUST NOT** classify players as “losers,” “lost,” or “failed” based on rank, resource totals, Lock-In timing, or participation state.

There is:
- No elimination
- No “game over”
- No failure screen
- No punitive status for low placement

### 9.2.2 System-Authored Language Prohibition (All Official Channels)

This is a hard design invariant.

All **system-authored** outcome/status copy and messaging:
- **MUST NOT** use “loss,” “defeat,” or “failure” language to label **player outcomes**
- **MUST** use neutral outcome terms (placement, rank, standing, choices, participation, eligibility, recorded outcomes, conversion secured)

Scope of “system-authored”:
- Includes **in-client** UI labels/buttons/cards/tutorials/notifications/system messages.
- Includes **out-of-client official communications**, including patch notes, marketing pages, emails, and push notifications.

Non-scope:
- **User-generated text** (chat messages, usernames, other user-entered strings) is unrestricted by this rule.
- **Admin tools and internal developer documentation** are not bound by this prohibition.

Technical exception (system issues):
- In official communications, technical “error/failed/failure” wording is allowed **only** when describing **system issues** (requests, outages, bugs, incidents). It **MUST NOT** be used to label player outcomes.

Action-error exception (rejected/denied actions):
- Action-error reason-code copy is **exempt** from this prohibition, but it **MUST** avoid phrasing that frames the player’s overall outcome/status as failure.
- Preferred action-error vocabulary: **rejected**, **denied**, **unavailable**, **not eligible**, **blocked**, **cannot complete**.

## 9.3 Valid Success States

Success in Too Many Coins is multi-modal and player-defined.
The system MUST treat the following as valid success outcomes.

1. **Winning placement is success**  
   Podium finish at natural end is success.

2. **Participating without top placement is normal**  
   Placement is not required for “success.” Most participants will not finish on the podium; this is expected and non-negative.

3. **Spending Seasonal Stars for power is a legitimate choice**  
   Dropping rank to buy Sigils / gain temporary power is a valid strategic decision, not a negative status.  
   Official messaging MUST NOT frame this as “sacrificing rank,” “throwing,” “griefing,” “trolling your placement,” “sabotaging your season,” “wasting stars.”

4. **Lock-In early is a valid strategic outcome**  
   Including forgoing end-of-season bonuses: **securing** conversion earlier is legitimate and must not be framed as a mistake or loss.

5. **Ending with low or zero Seasonal Stars is not failure**  
   This reflects timing, choices, and competition; it does not create a negative classification.

6. **Ending with zero Global Stars in the year is not failure**  
   If the player’s year-total is 0, the system SHOULD use neutral display such as: **“Global Stars this year: 0”** (no “progress” judgment language).

7. **Re-entry after exit is not failure**  
   Exiting and re-entering (fresh start) is a valid choice and must not be framed as a penalty or defeat.  
   The fact that the latest authoritative snapshot supersedes earlier ones SHOULD remain implicit in normal outcome copy (see help/FAQ surfaces for mechanics explanations).

8. **Idle status is not a loss**  
   Being Idle (or being idled) is a participation state that can reduce competitive outcomes, but it does not confer “failure,” “defeat,” or negative identity labeling.

## 9.4 Non-Failure Outcomes (Binding Framing Rules)

Outcomes that MUST be framed as non-failure:

- Low placement
- Late entry
- Early Lock-In
- Cosmetic-first play
- No podium finishes
- Minimal or zero year-total Global Stars
- Time spent spectating rather than participating
- Being Idle or temporarily disconnected

### 9.4.1 No Shame Framing

Official messaging MUST NOT:
- shame players for non-competitive decisions
- imply a single “correct” play style
- frame safety/preservation as cowardice
- frame spending resources as “wasting”

### 9.4.2 Canonical Participation/Status Labels (UI Tone Mapping)

When the UI uses status labels, it MUST use canonical labels.

#### 9.4.2.1 Stack Rule (Max 2)

Status labels are **stackable chips/badges**.

This rule applies to any single **status chip strip** UI region (e.g., the global header). If a surface has multiple distinct regions that each render status chips, the rule applies **per region**.

Per status-chip region:

Max displayed at once: **2**, ordered:

1. **Participation mode** (exactly one): `Participating` / `Spectating` / `Not participating`
2. **Constraint state** (at most one): `Disconnected` or `Idle`

Constraint precedence:
- `Disconnected` **overrides** `Idle` (if disconnected, do not also show idle).

Scope rule:
- When the region is **header-level**, the **Participation mode** chip MUST reflect **GlobalParticipationStatus** (9.4.2.4).
- If a **season-context** region renders participation chips, it MAY reflect **ViewedSeasonParticipationStatus** (9.4.2.4), but it MUST NOT replace or contradict the header-level chip.

#### 9.4.2.2 Participation Mode Labels

Participation mode labels:
- **Participating**
- **Spectating**
- **Not participating**

#### 9.4.2.3 Constraint State Labels

Constraint state labels:
- **Idle** (activity enforcement)
- **Disconnected**

Idle enforcement label:
- Idle due to inactivity enforcement MUST be labeled: **“Idle (activity required)”**.

Spectator/preview label:
- Spectator/preview mode MUST be labeled consistently as: **“Spectating”**.

Paused label:
- “Paused” does **not** exist as a distinct player-facing status label and MUST NOT be introduced by UI copy.

#### 9.4.2.4 Scope: Global vs Viewed Season Participation Status

Both of the following label scopes exist:

- **GlobalParticipationStatus** (header-level; derived client field defined in Chapter 15 §15.11.5): “Participating in any season (yes/no).”
- **ViewedSeasonParticipationStatus** (season context UI): “Participating in the currently viewed season (yes/no).”

The semantic distinction MUST exist. `GlobalParticipationStatus` is the canonical derived-field name (Chapter 15 §15.11.5). `ViewedSeasonParticipationStatus` is a UI concept name only; its exact naming/casing in code is implementation-defined.

##### 9.4.2.4.1 Deriving the header-level Participation mode label (Global)

When a header-level status chip strip renders a Participation mode chip, it MUST be derived from **GlobalParticipationStatus** and **whether the player is currently viewing a season**:

- If `GlobalParticipationStatus == true`: label MUST be **Participating**.
- Else if the UI is currently in a season-view surface (i.e., a season is being viewed): label MUST be **Spectating**.
- Else: label MUST be **Not participating**.

This prevents viewed-season context from masking the player’s global participation state.

##### 9.4.2.4.2 Deriving a season-context Participation mode label (Viewed season)

If a season-context UI region renders a Participation mode label/chip for the **currently viewed season**, it MUST be derived from **ViewedSeasonParticipationStatus**:

- If `ViewedSeasonParticipationStatus == true`: label MUST be **Participating**.
- Else (viewing a season but not economically in it): label MUST be **Spectating**.

##### 9.4.2.4.3 Example (binding semantics, not UI layout)

A player participating in season **A** while viewing season **B**:

- Header-level label: **Participating** (GlobalParticipationStatus = true).
- Season B context label (if shown): **Spectating** (ViewedSeasonParticipationStatus = false).

### 9.4.3 Prohibited Framing

Official messaging **MUST NOT**:
- imply a player “wasted” their season by choosing safety, cosmetics, or non-competitive play,
- imply late entry is inherently inferior as a matter of identity or value,
- equate “not winning” with “losing.”

### 9.4.4 Terminology Note on “Failure States” (Cross-Chapter Reference)

Chapter 11 (“Edge Cases and Failure States”) uses “failure states” to describe **technical/system state-recovery failures** (disconnects, recovery, edge conditions), not player-outcome “failure.”
This chapter’s “no failure condition” invariant refers strictly to **player outcomes and labeling**, not technical resilience scenarios.
