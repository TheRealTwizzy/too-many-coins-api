# Chapter 05 — Economy Design

This chapter defines the authoritative **seasonal economy model**, including:

- Economic Sources (UBI-only Coins)
- Economic Sinks (resource destruction)
- Scarcity Enforcement (finite inventories / no replenishment constraints)
- Inflation Controls (inflation dampening + hoarding suppression + modifiers)
- Trading Rules and Friction (valuation + fee computation)
- Economic Safety Constraints (floors/caps, determinism, overflow/underflow safety)
- Minimal economy telemetry/audit interface requirements (Chapter 15 owns final schema)

This chapter **does not** define:
- Tick order, buffering, intake sequencing, expiration sequencing (Chapter 03)
- Trade lock acquisition timing / lock semantics (Chapter 02)
- Participation predicates and lifecycle transitions (Chapter 03 / Chapter 14)
- Cosmetics or global-meta economies (Chapter 04 / outside seasonal economy)

**Lifecycle ordering:** `Expiration → Yearly reset → Phase change effective next tick`  
This ordering is owned by Chapter 03 and Chapter 14 and is not redefined here.

---

## 5.1 Vocabulary, Tick Naming, and Surface Concepts

### 5.1.1 Tick naming (hard rule)

All tick references in this chapter use **`season_tick_index`**.  
This chapter MUST NOT use ambiguous `tick_index`.

### 5.1.2 Two authoritative “surfaces”

This chapter relies on two authoritative value sources:

1) **`math_surface(season_tick_index = T)`**  
- Recomputed deterministically from **post-action** state for tick `T`.  
- Sampled **after** tick-`T` player actions resolve and **before** UBI is applied (Chapter 03).  
- Used only for tick-`T` UBI inputs (inflation/hoarding).

2) **Published acceptance surface (`surface_version = V`)**  
- Published at end of tick `V−1` and used during tick window `V` (Chapter 03).  
- Authoritative for acceptance/quotes.  
- **Locked quote rule:** for any quote-dependent action with `intake_tick = V`, the server MUST use `surface_version_used = V`, compute a locked quote, and reserve the full quoted cost at acceptance (Chapter 03).

### 5.1.3 Mapping: which formulas use which source

**UBI inputs**
- `inflation_factor_fp(T)` uses: `math_surface(T).season_total_coins_supply`
- `hoarding_factor_fp(P,T)` uses: `math_surface(T).player_awspd_window_coins_per_tick(P)` + season config `target_spend_rate_coins_per_tick`
- `base_ubi_*` uses: per-season economy config (this chapter)
- UBI branch selection uses: tick snapshot `activity_state_current(P)` as-of UBI phase start (Chapter 03)
- `sum_mod_fp(P,T)` uses: tick snapshot (post-action, pre-UBI modifier components for tick `T`, clamped per this chapter)

**Acceptance quoting**
- StarPrice uses: published `surface_version_used = intake_tick` field `star_price_coins_per_star`
- Vault cost uses: Chapter 04 published `VaultCostStars_surface[V][tier]` at acceptance tick `V`
- Trade valuation uses: published `surface_version_used = intake_tick` fields:
  - `star_price_coins_per_star`
  - `VaultCostStars_surface[V][tier]` (with deterministic sell-out fallback in 5.13.6)

**Trade fees**
- `trade_fee_rate_fp` uses: per-season economy config tier table (this chapter)

## 5.2 Economic Sources (UBI Only)

### 5.2.1 Coins mint sources (authoritative)

**Coins are minted only via UBI.** No other mint source exists.

Explicitly disallowed (non-exhaustive):
- Join stipends
- Tutorial grants
- Admin grants
- Refund mints
- Conversion mints
- Trade minting
- Event minting

Coins enter the seasonal economy exclusively through UBI during active season ticks.  
This is a non-negotiable invariant.

### 5.2.2 Remediation exception (deterministic re-derivation only)

Operational remediation MAY adjust state **only** via a deterministic **re-derivation job** that recomputes canonical state from authoritative logs/snapshots.

Constraints:
- Remediation MUST NOT be ad-hoc mint/burn.
- Remediation MUST emit immutable audit events describing the remediation run identity, inputs, and derived diffs.
- Remediation MUST produce identical results on deterministic replay.

---

## 5.3 Economic Sinks vs Resets (Resource Destruction Accounting)

### 5.3.1 Definitions

**Sink:** any **in-season resource destruction** event (Coins, SeasonalStars, Sigils, etc.).  
Examples:
- Coins burned on SeasonalStar purchases
- Coins burned as trade fees
- SeasonalStars burned on Sigil vault purchases (Chapter 04 purchase action)
- Sigils burned on Boost purchases (Chapter 03 boost purchase action)

**Reset destruction:** lifecycle- or season-bound destruction/clears (e.g., Lock-In destruction, expiration destruction).  
Resets are **not sinks**, but they are still **resource destruction** and MUST be logged as such (tagged as reset destruction).

### 5.3.2 Telemetry tagging requirement (interface only)

All sinks MUST be logged as economic destruction events (Chapter 15).  
All reset destructions MUST be logged as economic destruction events **tagged as reset** (Chapter 15).

## 5.4 Determinism and Numeric Rules

### 5.4.1 Types

- Coins: `int64`
- SeasonalStars: `int64`
- GlobalStars: `int64`
- Fixed-point: `int32`/`int64`, where `1_000_000 = 1.0`

All intermediate math MUST use `int128` (or equivalent wide intermediate).

### 5.4.2 Fixed-point multiply rule (floor after each step)

For a step applying multiplier `mult_fp`:

`result = floor( base * mult_fp / 1_000_000 )`

Rules:
- Use `int128` intermediates
- Apply `floor` at each step (not only at the end of a chain)
- Clamp/store back to `int64` only after safe-range validation

### 5.4.3 Overflow/underflow safety

- Player actions MUST be rejected at acceptance time if they would overflow/underflow.
- System operations (UBI, conversions) MUST validate immediately prior to commit; if invalid, MUST NOT apply and MUST emit audit/telemetry.
- Balances MUST NOT become negative.

### 5.4.4 Int128 runtime overflow guard (trade valuation)

If any intermediate multiply/add would overflow signed `int128`, the action MUST be rejected at **acceptance** with `reason_code=numeric_overflow` (no state changes, no escrow created).

(Chapter 15 owns the closed reason-code set and MUST include `numeric_overflow`.)

## 5.5 Per-Season Economy Configuration (Ownership + Immutability + Validation)

### 5.5.1 Immutability

The per-season economy configuration MUST be immutable once the season starts. Any change requires a new season.

### 5.5.2 Required per-season economy config fields (Chapter 05)

- **UBI**
  - `base_ubi_active_per_tick:int64`
  - `base_ubi_idle_factor_fp:int32`
  - `ubi_min_per_tick:int64`

- **Inflation dampening**
  - `inflation_table: list[(x:int64, factor_fp:int32)]`
  - `inflation_min_factor_fp:int32`

- **Hoarding suppression**
  - `hoarding_window_W_ticks:int32`
  - `target_spend_rate_coins_per_tick:int64`
  - `hoarding_min_factor_fp:int32`

- **StarPrice**
  - `starprice_table: list[(m:int64, price_coins_per_star:int64)]`
  - `star_price_cap_coins_per_star:int64`

- **Trade fees**
  - `trade_fee_tiers: list[(threshold_value_coins:int64, fee_rate_fp:int32)]`
  - `trade_min_fee_coins:int64`

- **Modifier clamps (per-component)**
  - `modifier_component_clamps: list[(modifier_id:uint32, min_fp:int32, max_fp:int32)]`
  - `sum_mod_fp_min:int32` (required; default `-1_000_000`)
  - `mod_cap_multiplier_fp:int32` (required; e.g., default `5_000_000`)

**Authority note:** `modifier_id:uint32` values are a **global fixed enum registry** (closed; stable; never reused). (Defined/owned by Chapter 03/15; referenced here.)

**Sigil vault scarcity and pricing** are owned by **Chapter 04**. Chapter 05 references those surfaces for valuation only.

### 5.5.3 Required config validations (season start MUST fail if invalid)

- Tables MUST be non-empty.
- Keys strictly increasing:
  - `inflation_table.x` strictly increasing
  - `starprice_table.m` strictly increasing
  - `trade_fee_tiers.threshold_value_coins` strictly increasing
- Inflation/hoarding fixed-point bounds:
  - `inflation_table.factor_fp ∈ [0, 1_000_000]`
  - `inflation_min_factor_fp ∈ [1, 1_000_000]`
  - `hoarding_min_factor_fp ∈ [1, 1_000_000]`
- Floors/caps consistent:
  - `star_price_cap_coins_per_star >= max(starprice_table.price_coins_per_star)` and `>= 1`
  - `all starprice_table.price_coins_per_star >= 1`
  - `ubi_min_per_tick >= 1`
  - `target_spend_rate_coins_per_tick > 0`
  - `trade_min_fee_coins >= 0`
- Trade fee tiers MUST have a base tier:
  - `trade_fee_tiers[0].threshold_value_coins == 0` (mandatory)
- Modifier clamp value bounds:
  - For each clamp row: `min_fp <= max_fp`
  - `min_fp ∈ [-1_000_000, +4_000_000]`
  - `max_fp ∈ [-1_000_000, +4_000_000]`
- Sum modifier bounds:
  - `sum_mod_fp_min == -1_000_000` (mandatory)
  - Define `sum_mod_fp_max = mod_cap_multiplier_fp - 1_000_000`
  - Require `mod_cap_multiplier_fp >= 0` and `mod_cap_multiplier_fp <= 5_000_000`
- Modifier clamp table coverage:
  - For the season’s enabled boost/system-event set (Chapter 03), every possible `modifier_id` that can contribute to `sum_mod_fp` MUST exist in `modifier_component_clamps`.
  - Missing clamp coverage MUST prevent season start.

- `player_spend_window_sum_coins(P):int64`

- `player_hoarding_factor_fp(P):int32`  
  Output of 5.9 for `P` on tick `T` (optional cache; MUST equal the deterministic formula)

Optional (not economy-driving unless referenced by some other chapter):
- `season_day_index:int32`

## 5.7 UBI Minting Model (Coins)

### 5.7.1 Eligibility (aligns to Chapter 03)

UBI eligibility is snapshot-gated by Chapter 03:

`ubi_eligible_snapshot(P,T)` is authoritative.

If `ubi_eligible_snapshot(P,T) == false`, the player receives `UBI = 0` on tick `T`.

### 5.7.2 Same-tick suppression (terminal actions)

If `lock_in_confirm` resolves for the player on tick `T`, the player MUST receive `UBI = 0` on tick `T` (Chapter 03).  
This suppression overrides `ubi_min_per_tick`.

### 5.7.3 Join timing (explicit alignment)

If `season_join` resolves on tick `T`, it does not create UBI eligibility on tick `T`. Eligibility begins at snapshot of `T+1` (Chapter 03).

### 5.7.4 Base UBI by activity branch (timing aligns to Chapter 03)

If `ubi_eligible_snapshot(P,T) == true` and Lock-In suppression does not apply:

- `base_ubi_active = base_ubi_active_per_tick`
- `base_ubi_idle = floor(base_ubi_active_per_tick * base_ubi_idle_factor_fp / 1_000_000)`

Branch selection uses `activity_state_current(P)` as it exists at the start of the UBI phase (after action resolution, before activity evaluation):

- Action-driven changes before UBI (e.g., `idle_modal_ack` on tick `T`) MAY affect tick-`T` branch selection.
- Inactivity-triggered Active→Idle transitions occur after UBI and therefore apply starting `T+1`.

### 5.7.5 UBI computation order (floor after each step)

If payable:

1) Start with base (`active` or `idle`)  
2) Apply inflation dampening factor (5.8)  
3) Apply hoarding suppression factor (5.9)  
4) Apply modifiers (5.10)  
5) Apply minimum floor: `ubi = max(ubi, ubi_min_per_tick)`  
6) Commit mint: `Coins += ubi`

All multiplies MUST use `int128`, and MUST `floor` after each step.

## 5.8 Inflation Dampening (UBI Control)

### 5.8.1 Supply metric `season_total_coins_supply` (math_surface input)

`season_total_coins_supply` includes:
- all player Coin balances (including Offline players still joined)
- all Coin escrow/reserved/locked amounts for pending actions/trades
- any other Coin escrow state that is still part of season supply

It is net of all destruction as-of post-action resolution for tick `T`.

### 5.8.2 Inflation factor formula (piecewise-linear)

Given per-season config control points:
`inflation_table = [(x0,f0_fp), …, (xN,fN_fp)]`

Input:
`x = math_surface(T).season_total_coins_supply`

Clamp input:
- if `x <= x0` → `factor_fp = f0_fp`
- if `x >= xN` → `factor_fp = fN_fp`
- else find `xi <= x < x{i+1}` and compute:

`factor_fp = floor( f_i + ((f_{i+1}-f_i)*(x-x_i)) / (x_{i+1}-x_i) )`

Rules:
- Use `int128` intermediates.
- If `x == x{i+1}`, the result MUST be `f{i+1}`.
- Clamp final:
  `factor_fp = clamp(factor_fp, inflation_min_factor_fp, 1_000_000)`

### 5.8.3 Apply inflation to UBI (floor-after-step)

`ubi = floor( ubi * factor_fp / 1_000_000 )`

---

## 5.9 Hoarding Suppression (UBI Control)

### 5.9.1 Spend definition for hoarding suppression

`spend(P,t)` includes **only** explicit **Coins burned by a player action** on tick `t`.

Included:
- Coins burned on SeasonalStar purchases
- Coins burned as trade fees

Excluded:
- Coins destroyed by resets (Lock-In destruction, expiration destruction, lifecycle wipes)
- Transfers (trades moving Coins between players)

### 5.9.2 Windowing

Let `W = hoarding_window_W_ticks`.

For tick `T`:
- `player_spend_window_sum_coins(P,T) = Σ spend(P,t)` for `t ∈ [T-W, T-1]`

Window semantics:
- Missing pre-season ticks are treated as `0` spend.
- Non-participation periods are treated as `0` spend.
- On season re-entry (exit then re-enter same season), the spend window MUST reset **immediately on the re-entry resolution tick** (so `math_surface(T)` for that tick sees a reset window).

Define:
- `AWSPD(P,T) = floor(player_spend_window_sum_coins(P,T) / W)` (coins per tick)

### 5.9.3 Hoarding factor formula (never boosts)

Let season config:
- `target = target_spend_rate_coins_per_tick` (must be > 0)

Compute:
- `raw_fp = floor( AWSPD(P,T) * 1_000_000 / target )` using `int128`
- `factor_fp = clamp(raw_fp, hoarding_min_factor_fp, 1_000_000)`

Apply:
`ubi = floor( ubi * factor_fp / 1_000_000 )`

## 5.10 Boost/System Modifiers (UBI Control)

### 5.10.1 Per-component clamp rules (config-owned)

For tick `T`, the server constructs modifier components for player `P` as a list of:
`(modifier_id:uint32, component_fp:int32)`.

For each component:
- Lookup `(min_fp,max_fp)` for `modifier_id` in `modifier_component_clamps`.
- Clamp: `component_fp_clamped = clamp(component_fp, min_fp, max_fp)`.

Sum:
- `sum_mod_fp_raw = Σ component_fp_clamped` using `int128` accumulation.

Clamp sum (required):
- `sum_mod_fp = clamp(sum_mod_fp_raw, sum_mod_fp_min, sum_mod_fp_max)`
- Where:
  - `sum_mod_fp_min = -1_000_000`
  - `sum_mod_fp_max = mod_cap_multiplier_fp - 1_000_000`

Final multiplier:
- `mult_fp = clamp(1_000_000 + sum_mod_fp, 0, mod_cap_multiplier_fp)`

### 5.10.2 Apply modifiers (floor-after-step)

`ubi = floor( ubi * mult_fp / 1_000_000 )`

## 5.11 StarPrice Curve (Coins → SeasonalStars)

### 5.11.1 Published quote field (authoritative)

Acceptance-time StarPrice uses:
- `star_price_coins_per_star:int64` from published `surface_version_used = intake_tick`

This value MUST be logged as `surface_version_used` for the purchase.

### 5.11.2 `season_total_coins_supply_end_of_tick` (published-surface input)

Definition:  
`season_total_coins_supply_end_of_tick(t)` is the total Coins existing in season state **after all tick-`t` phases complete** (including UBI mint, all burns, transfers, resets, and escrow updates), sampled at the tick-end boundary immediately before publishing `surface_version=t+1`.

Inclusions (MUST include):
- escrow/reserved Coins
- free/unreserved player balances
- anything “pending” represented as escrow/reserve within season state

### 5.11.3 StarPrice publishing input timing (explicit)

Published `surface_version = V` is computed at end of tick `V−1` (Chapter 03).  
StarPrice for `surface_version = V` MUST be computed using:

- `m = season_total_coins_supply_end_of_tick(V−1)` (post-UBI, end-of-tick supply)

### 5.11.4 StarPrice function (piecewise-linear + cap)

Config:
`starprice_table = [(m0,p0), …, (mN,pN)]`

Interpolation is piecewise-linear with `floor` and `int128` intermediates (same as 5.8).

After interpolation:
- `p = min(p, star_price_cap_coins_per_star)`
- require `p >= 1`

Publish:
- `star_price_coins_per_star = p` into `surface_version = V`.

### 5.11.5 Purchase contract: `coins_to_spend` (locked quote + “spend up to”)

Request:
- `coins_to_spend:int64` (must be `> 0`)

At acceptance:
- Lock `surface_version_used = intake_tick`
- Read `price = star_price_coins_per_star` from that surface
- `stars_out = floor(coins_to_spend / price)`
- If `stars_out < 1`, reject
- `coins_spent = stars_out * price`
- Affordability requirement: require `Coins >= coins_spent` (NOT `Coins >= coins_to_spend`)
- Reserve `coins_spent` at acceptance (locked quote rule)
- No reservation exists for the remainder; request means “spend up to `coins_to_spend`”

At commit:
- Burn `coins_spent` (Coins sink)
- Credit `SeasonalStars += stars_out`
- No repricing; commit MUST use the locked quote.

## 5.12 Scarcity Enforcement (Sigils and Finite Inventory)

### 5.12.1 Sigil vault scarcity and pricing (Chapter 04-owned)

The seasonal Sigil Vault (Chapter 04):
- has fixed inventory at season start,
- is globally shared,
- is not replenished.

Pricing is defined by Chapter 04 as step tables over remaining inventory:
- `VaultCostStars_surface[T][tier]`
- If `vault_remaining_snapshot(T)[tier] == 0`, tier is not purchasable and has no surface price (Chapter 04)

### 5.12.2 Disabled tiers (season-wide)

If a tier is disabled for a season (e.g., configured supply = 0 in Chapter 04’s season config):
- It MUST NOT appear in player inventories.
- It MUST NOT be purchasable.
- It MUST NOT be tradable.
- Any trade referencing it MUST be rejected with `reason_code=invalid_trade_asset`.

This makes “no fallback price exists” non-reachable.

### 5.12.3 Resource destruction semantics (sink classification)

- SeasonalStars spent on `purchase_sigils_vault` are **destroyed** (sink).
- Sigils spent on `purchase_boost` are **destroyed** (sink).

(Chapter 04/03 own the actions; Chapter 05 owns sink classification for accounting.)

## 5.13 Trading Rules and Friction (Valuation + Fees)

### 5.13.1 Allowed trade contents (closed set)

Trades may include only:
- Coins
- Sigils (by tier multiset)

Trades MUST NOT include SeasonalStars or GlobalStars.

Trade composition constraints (binding; owner = Chapter 05):
- Coins ↔ Coins trades MUST NOT exist.
- One-sided trades (gifting) MUST NOT exist.
- Both players MUST contribute at least one positive value component.
- Validation predicate (canonical):

Let `A = (A_coins, A_sigils_count)` and `B = (B_coins, B_sigils_count)` where:
- `*_coins:int64`
- `*_sigils_count:int32` (sum of `sigil_count_by_tier` across tiers)

```
valid :=
  ( (A_coins > 0 OR A_sigils_count > 0) AND
    (B_coins > 0 OR B_sigils_count > 0) AND
    NOT (A_sigils_count == 0 AND B_sigils_count == 0) )
```

Where the final clause enforces “no Coins-for-Coins.”

### 5.13.2 Canonical trade model (economic payload contract)

`trade_initiate` MUST specify the full deal, including:
- `trade_id:uint64` (client-supplied; must be unique within `(season_id, trade_id)`)
- both sides’ offered assets:
  - `sideA.coins_amount:int64`
  - `sideA.sigil_count_by_tier:[5]int32`
  - `sideB.coins_amount:int64`
  - `sideB.sigil_count_by_tier:[5]int32`
- the intended counterparty identity (owned by Chapter 03’s social/mechanics layer)

`trade_accept` only confirms acceptance of the existing deal identified by `(season_id, trade_id)`.

Quote/fee locking moment:
- Valuation and fee for a trade are locked at **`trade_initiate` acceptance** and stored on the offer.
- `trade_accept` MUST use the stored locked quote/fee (no repricing).

### 5.13.3 Identity and conflict replacement

Canonical trade identity key:
- `(season_id, trade_id)`

Within the same `intake_tick`, actions for the same `(season_id, trade_id)` are processed in deterministic order `(intake_tick, accept_seq)`.

`accept_seq` is the canonical monotonic global acceptance ordering key (defined in Chapter 03).

Rule:
- The **first applied terminal transition** for the identity wins.
- Later transitions in the same tick window MUST be rejected with `conflict_replaced`.

### 5.13.4 Reservation timing (initiate vs accept)

- At `trade_initiate` acceptance:
  - Lock `surface_version_used = intake_tick` (the intake tick of this `trade_initiate`).
  - Compute and store on the offer:
    - `declared_trade_value_coins` (per 5.13.6–5.13.7, using `surface_version_used`)
    - `locked_trade_fee_coins` (per 5.13.8, using the same `declared_trade_value_coins`)
  - Reserve/escrow (initiator side):
    - initiator offered assets (Coins + Sigils)
    - initiator fee in Coins = `locked_trade_fee_coins`
  - Offer becomes OPEN.

- At `trade_accept` acceptance:
  - MUST use the offer’s stored `surface_version_used`, `declared_trade_value_coins`, and `locked_trade_fee_coins` (no repricing).
  - Reserve/escrow (acceptor side):
    - acceptor offered assets (Coins + Sigils)
    - acceptor fee in Coins = `locked_trade_fee_coins`
  - If any acceptor reservation/affordability/validity fails, reject `trade_accept` with no side effects:
    - No swap occurs.
    - Any temporary acceptor holds MUST be released before responding.
    - The offer remains OPEN and initiator escrow remains locked until cancel/decline/expiry/success.
  - On success: perform the swap + fee burn atomically.

### 5.13.5 Declared trade value locking moment

Trade valuation and fees are locked at **`trade_initiate`**:
- `surface_version_used = intake_tick` for the initiate action
- `declared_trade_value_coins` and `locked_trade_fee_coins` are computed once from that surface and stored on the offer.
- `trade_accept` and trade commit MUST use the stored locked quote/fee, with no repricing.

### 5.13.6 Sigil valuation in Coins (vault-cost-derived, sell-out fallback)

Let `V = surface_version_used`.

From published surface `V`:
- `star_price = star_price_coins_per_star:int64`
- `vault_cost_stars = VaultCostStars_surface[V][tier]` if defined

Sell-out fallback (deterministic):
- If `VaultCostStars_surface[V][tier]` is undefined due to sell-out at tick window `V`,
  then use the **last published non-null** vault cost for that tier:
  - Let `V*` be the maximum `surface_version <= V` such that `VaultCostStars_surface[V*][tier]` is defined.
  - Use `VaultCostStars_surface[V*][tier]`.

Sigil value in Coins:
- `sigil_value_coins[tier] = sigil_count[tier] * vault_cost_stars[tier] * star_price`

Rules:
- Use `int128` intermediates.
- If any intermediate overflows signed `int128`, reject at acceptance with `reason_code=numeric_overflow`.
- Validate the final `value_side` and `declared_trade_value_coins` fit `int64`; otherwise reject at acceptance with `reason_code=numeric_overflow`.

Side value:
- `value_side = coins_amount + Σ_tier sigil_value_coins[tier]`

Declared trade value:
- `declared_trade_value_coins = value_sideA + value_sideB`

### 5.13.7 Tiered fee rate selection (deterministic)

Config:
`trade_fee_tiers = [(threshold0, rate0_fp), …]`, strictly increasing thresholds, and **must** satisfy:
- `threshold0 == 0`

Select:
- choose the tier with the highest `threshold <= declared_trade_value_coins`.

### 5.13.8 Fee formula (symmetric, both parties, burned)

For each party:
- `fee_raw = floor(declared_trade_value_coins * fee_rate_fp / 1_000_000)` using `int128`
- `fee = max(trade_min_fee_coins, fee_raw)`

Both parties pay the same computed `fee`. Fees are burned (Coins sink).

### 5.13.9 Post-initiate lifecycle actions (cancel/decline)

`trade_decline`
- burn_fee: **no**
- release initiator escrow: **yes** (only if trade not yet accepted)
- release acceptor escrow: **no** (no acceptor escrow exists pre-accept)
- state: mark terminal `DECLINED` (cannot be accepted afterward)

`trade_cancel`
- burn_fee: **no**
- release initiator escrow: **yes** (only if trade not yet accepted)
- release acceptor escrow: **no** (no acceptor escrow exists pre-accept)
- state: mark terminal `CANCELED` (cannot be accepted afterward)

If a cancel/decline arrives **after** a valid accept has been applied for the same identity, it MUST be rejected as a conflict (`conflict_replaced`) and MUST NOT be processed.

### 5.13.10 Post-accept invalidation causes (closed set)

After a successful `trade_accept` acceptance, a trade can fail only due to:

- `expiration_tick`
- `season_expired`
- `mode_disallows_action`
- `stale_client`
- `conflict_replaced`

`insufficient_balance` and `insufficient_assets` MUST be **acceptance-time only** rejects (never post-accept), due to full reservation at `trade_initiate` (initiator side) and `trade_accept` (acceptor side).

## 5.14 Conversion Accounting (SeasonalStars → GlobalStars)

### 5.14.1 Lock-In conversion

On successful Lock-In (Chapter 03):
- `GlobalStars += SeasonalStars_current`
- `SeasonalStars = 0`

Always 1:1.

### 5.14.2 Natural season-end conversion (eligible end-finishers only)

On natural season end:
- If `eligible_end_snapshot(P,S)` is true (Chapter 03), then:
  1) Apply expiration bonuses first (Chapter 06)
  2) Convert:
     - `GlobalStars += SeasonalStars_current`
     - `SeasonalStars = 0`

Always 1:1.

### 5.14.3 Multiple Lock-Ins (anti-double-count note)

Each Lock-In converts only the player’s current SeasonalStars at that moment. Because SeasonalStars is set to 0 after conversion, double-counting MUST NOT occur.

## 5.15 Economic Safety Constraints (Floors, Caps, Determinism)

### 5.15.1 Floors and caps (hard constraints)

The economy MUST enforce:

- `ubi_min_per_tick >= 1` for payable UBI (except Lock-In suppression tick, which forces 0)
- `inflation_min_factor_fp` (minimum dampening factor)
- `hoarding_min_factor_fp` (minimum hoarding factor)
- `star_price_cap_coins_per_star` (maximum StarPrice)
- `mod_cap_multiplier_fp` (maximum final multiplier)

These constraints exist to prevent mathematical lockout and preserve late-season viability.

### 5.15.2 Late-season viability KPI (tuning goal)

Tuning SHOULD satisfy:

- A fresh joiner at **day 21** (start of day 22), playing Active ~**1 hour/day**, can typically buy **≥ 100 SeasonalStars** before season end.

This is a **tuning KPI**, not a runtime enforcement rule.

### 5.15.3 Deterministic replay requirements

- All economic computations MUST be deterministic given authoritative logs/snapshots and immutable season config.
- All quote-dependent actions MUST log:
  - `surface_version_used`
  - locked quote amounts (costs/fees/stars_out where applicable)

## 5.17 Non-Negotiable Invariants

- Coins are minted **only** via UBI.
- Remediation is allowed only via deterministic re-derivation (no ad-hoc mint/burn).
- Sinks and reset destructions are logged as resource destruction (reset-tagged where applicable).
- Inflation dampening and hoarding suppression are deterministic and have non-zero minimum factors.
- Hoarding suppression factor MUST NOT exceed `1_000_000` (never boosts UBI).
- StarPrice is deterministic and hard-capped; price MUST be `>= 1`.
- No repricing after acceptance; acceptance locks `surface_version_used`, commit uses it.
- Trades are Coins+Sigils only; valuation is vault-cost-derived with deterministic sell-out fallback; fees are tiered, symmetric, burned; reservation timing is explicitly defined.
- Disabled Sigil tiers MUST NOT be tradable.
- Conversion is always 1:1 and sets SeasonalStars to 0 after credit.
- Numeric behavior is deterministic: `int128` intermediates, `floor` where specified, strict overflow/underflow protection.
