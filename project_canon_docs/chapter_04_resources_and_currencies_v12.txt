# Chapter 04 — Resources and Currencies

This chapter defines the authoritative properties, lifecycle behavior, deterministic ordering guarantees (resource-scoped), visibility rules, and invariants of all resource types.

This chapter defines **resource behavior only**.  
Economic pricing and trade friction formulas are defined in **Chapter 05**.  
Boost mechanics and effect semantics are defined in **Chapter 03**.  
Season timing, expiration semantics, and blackout windows are defined in **Chapter 07**.  
Authoritative per-tick phase order (including where Sigil drop evaluation occurs) is defined in **Chapter 03**. This chapter MUST NOT define tick order.

If a behavior is not explicitly defined here, it does not exist.

---

## 4.1 Resource Classes

The system contains five resource classes:

1. **Coins** (Seasonal Currency)  
2. **Seasonal Stars**  
3. **Global Stars**  
4. **Sigils** (Seasonal Strategic Resources)  
5. **Cosmetic / Meta Unlocks**

Each class has explicit rules governing creation, destruction, persistence, deterministic validation constraints, visibility, and edge-case handling.

### 4.1.1 Shared Numeric Domain and Validation Rules

**Numeric domain.**
- Coins, Seasonal Stars, and Global Stars MUST be stored as **signed int64**.
- These values MUST be treated as **integers only** (no fractional values, no hidden decimal storage).
- These values MUST NOT be negative.

**Int64-domain validation (hard constraint).**
- Any operation (player action or system operation) that would result in a value outside the valid int64 range MUST be rejected **before commit**.
- For **player actions**, this rejection MUST occur at **acceptance time** (per Chapter 03 acceptance rules), preserving “cannot fail after acceptance” for insufficiency/overflow cases.
- For **system operations** (e.g., UBI accrual, expiration bonuses, conversion), validation MUST occur immediately prior to state commit in that phase; if invalid, the state change MUST NOT be applied and the system MUST continue running (recoverable, non-fatal). Audit/telemetry expectations are defined in Chapter 15.

**No design cap.**
- No artificial “design maximum cap” exists for any numeric resource.
- The only hard bound is the technical int64 domain constraint above.

### 4.1.2 UI Magnitude Shorthand (T+)

UI-only magnitude shorthand is defined in **Chapter 10**. This chapter makes no UI rendering requirements beyond resource visibility constraints.

### 4.1.3 Visibility Matrix (Authoritative)

| Resource       | Self | Others (participants) | Spectators | Where shown |
| -------------- | ---: | --------------------: | ---------: | ---------- |
| Coins          | Yes  | No                    | No         | Season view only (self HUD/UI) |
| Seasonal Stars | Yes  | Yes                   | Yes        | Season view (leaderboard + season UI) |
| Global Stars   | Yes  | Yes                   | Yes        | Global view (global leaderboard / profiles) |
| Sigils         | Yes  | No                    | No         | Season view only (self inventory UI) |
| Cosmetics      | Yes  | Profile: Yes / Leaderboard: No | Profile: Yes / Leaderboard: No | Global profiles only (not any leaderboard) |

## 4.2 Coins (Seasonal Currency)

### 4.2.1 Definition

Coins are the sole economic currency within a season.

Coins:
- MUST be integers only
- MUST NOT be fractional
- MUST NOT be negative
- MUST exist only within a season participation context
- MUST NOT persist beyond season exit (Lock-In or natural expiration only)
- MUST NOT transfer across seasons

UBI accrual produces integer Coin output only (flooring fixed-point computation per tick; see 4.2.2 and Chapter 05).

### 4.2.2 Creation

Coins are created exclusively through **UBI accrual**.

UBI computation MAY use fixed-point internal math, but the committed Coin increment per tick MUST be:
- floored to an integer each tick, and
- validated under int64-domain validation before commit (4.1.1)

Coins MUST NOT be:
- granted by admin action
- injected for testing (no exceptions in any environment)
- generated by trading
- generated by boosts
- generated by Sigils
- generated by system events

No alternate faucet exists.

### 4.2.3 Destruction

Coins are permanently destroyed when:
- purchasing Seasonal Stars
- paying trade fees
- Lock-In resolves (`lock_in_confirm` resolution; see Chapter 03 for ordering)
- season expiration resolves

Coins MUST NEVER convert into Global Stars or any cross-season resource.

### 4.2.4 Balance Constraints

- A Coin balance MUST NEVER be negative.
- No artificial cap exists (bounded only by int64-domain validation).

### 4.2.5 Visibility

- A player’s Coin balance MUST always be visible to that player while economically participating in a season.
- Coin balances MUST NEVER be visible to other players.
- Coins MUST NOT be displayed in spectator telemetry.

If a player participates in Season A while spectating Season B, their Season A Coin balance MUST remain visible to that player.

If a player is not economically participating in any season:
- no server-side Coin balance exists for that player, and
- no Coin counter MUST be displayed.

## 4.3 Seasonal Stars

### 4.3.1 Definition

Seasonal Stars represent competitive standing within a single season.

Seasonal Stars:
- MUST be integers only
- MUST NOT be fractional
- MUST NOT be negative
- MUST exist only within a season
- MUST NOT persist beyond season exit (Lock-In or natural expiration only)

Seasonal Stars determine seasonal leaderboard placement.

### 4.3.2 Creation

Seasonal Stars are obtained exclusively via:

1. **Purchasing using Coins** (pricing math is defined in Chapter 05).

No alternate creation path exists.

Note: End-of-season participation and placement bonuses are awarded as **Global Stars** during expiration finalization (Chapter 06) and MUST NOT modify Seasonal Stars.

### 4.3.3 Spending

Seasonal Stars MAY be spent only to purchase Sigils from the seasonal Sigil Vault.

Spending rules:
- Seasonal Star deduction MUST be **reserved at acceptance** and committed at resolution.
- Accepted purchases MUST NOT later fail due to insufficiency of Seasonal Stars.
- On resolution:
  - the player’s Seasonal Star total MUST decrease immediately,
  - the Sigil vault inventory MUST decrement (if applicable), and
  - the seasonal leaderboard MUST recalculate deterministically (leaderboard system is defined elsewhere; recalculation timing is governed by Chapter 03).

If insufficient Seasonal Stars are available at acceptance, the purchase MUST fail completely.
- No partial fulfillment is permitted.

### 4.3.4 Final Tick & Expiration Restrictions

Sigil purchases from the vault are invalid on:
- the final valid economic tick, and
- the expiration tick

Acceptance-time legality MUST reject these attempts.
- No Seasonal Stars are reserved or deducted.
- No Sigils are granted.

If a purchase would resolve on an invalid tick (as computed at acceptance), it MUST be rejected at acceptance (no queue/rollover behavior exists).

### 4.3.5 Conversion and Exit Effects

**No persistence beyond season exit.**
Seasonal Stars MUST NOT persist beyond season exit or season expiration finalization.

**Lock-In conversion timing (authoritative, resource-facing).**
When a player exits a season via a successful `lock_in_confirm` (Chapter 03), the server MUST:

- convert **all** current Seasonal Stars to Global Stars at **1:1 integer**:
  - `GlobalStars += SeasonalStars_current`
- then destroy Seasonal Stars:
  - set `SeasonalStars = 0` (defensive invariant)

This conversion is applied **immediately at Lock-In resolution** (not deferred).

**Lock-In snapshot (not a currency).**
On each successful Lock-In, the server records/overwrites an authoritative Lock-In snapshot for that season (Chapter 03), capturing (at minimum):

- Seasonal Stars at Lock-In (pre-conversion / pre-destruction)
- cumulative Participation Time total at Lock-In

The **latest** successful Lock-In snapshot before the Lock-In cutoff is authoritative for ranking/surfaces and end-of-season evaluation interfaces (Chapters 03 and 09). Lock-In itself grants **no** expiration bonuses (Chapter 03).

**Natural expiration finalization (resource-facing).**
During expiration finalization at tick `E` (Chapter 03):

- **Eligible end-finishers only** (`eligible_end_snapshot(P,S) == true`):
  1. apply end-of-season bonuses as Global Stars (Chapter 06),
  2. convert remaining Seasonal Stars → Global Stars at **1:1 integer**,
  3. set `SeasonalStars = 0`.

- **Non-eligible joined players** (`eligible_end_snapshot(P,S) == false`):
  - MUST receive **no** end-of-season bonuses and **no** natural-end conversion.
  - any remaining Seasonal Stars (if any) are destroyed during expiration cleanup.

Chapter 03 defines authority/eligibility and ordering; Chapter 06 defines bonus magnitudes.

## 4.4 Global Stars

### 4.4.1 Definition

Global Stars represent cumulative achievement across completed seasons within a yearly cycle.

Global Stars:
- MUST be integers only
- MUST NOT be fractional
- MUST NOT be negative
- MUST be non-transferable
- MUST be non-tradable

Global Stars determine global leaderboard rank.

Leaderboards MUST always use actual stored numeric values. UI shorthand displays MUST NEVER be used in ordering logic.

### 4.4.2 Creation

Global Stars are credited via exactly two mechanisms:

1. **1:1 conversion from Seasonal Stars** at:
   - Lock-In resolution, and
   - natural season expiration finalization

2. **Expiration bonuses** credited as Global Stars at **season expiration finalization only**, as defined in **Chapter 06**.
   - Ordering guarantee: at expiration finalization, expiration bonuses MUST be applied **before** the 1:1 SeasonalStars → GlobalStars conversion (see Chapter 13 ordering).

No other creation path exists.

### 4.4.3 Spending

Global Stars MAY be spent exclusively on **Cosmetic / Meta Unlocks**.

Spending rules:
- Spending MUST permanently reduce Global Star total.
- Spending MAY reduce global leaderboard rank.
- Spending is irreversible and MUST NOT be refundable.

Execution model (authoritative, resource-facing):
- A cosmetic purchase is a **global tick-governed action** (not season-scoped). It is accepted/buffered and committed on the next global 1-second tick boundary (Chapter 03).
- Cosmetic purchases MUST be serialized against the player’s tick commit boundary using the per-player serialized action queue described in Chapter 03.
- Phase order at any commit boundary is fixed:
  1. **Season-tick actions** (including `lock_in_confirm`)
  2. **Global actions** (including cosmetic purchases)
- Therefore, if `lock_in_confirm` and a cosmetic purchase are both pending for the same commit boundary, the Lock-In credit is applied **before** the cosmetic spend is evaluated.

Commit behavior:
- The server MUST validate:
  - SKU validity / ownership constraints (this chapter; progression constraints in Chapter 06 if applicable)
  - `GlobalStars` balance sufficiency
- If sufficient, the server MUST atomically:
  - deduct cost from `GlobalStars`
  - grant/unlock the cosmetic entitlement
  - update global leaderboard views as required (4.4.4)

Tick restrictions (context-gated, deterministic):
- Cosmetic purchases MUST be rejected if their commit boundary would occur while the **yearly reset** is executing (4.4.5).
- Any attempted purchase during that window MUST be rejected (no reservation, no deduction, no unlock granted).
- Rejection MUST use `reason_code=yearly_reset_in_progress` (Chapter 15).

No other special tick restrictions exist for cosmetic purchases. Outside the yearly reset execution window, cosmetics follow normal gating (ServerMode + feature-gates, including `gate_cosmetics`).


### 4.4.4 Leaderboard Update Timing

When Global Stars are spent:
- the global leaderboard MUST reflect the new stored total immediately as part of the same committed transaction.
- seasonal leaderboard calculations MUST remain independent.

Global and seasonal leaderboards are separate systems.

### 4.4.5 Yearly Reset

Yearly reset effects:
- erase all Global Star totals,
- reset global leaderboard rank,
- permanently destroy unspent Global Stars,
- execute atomically,
- preserve Cosmetic unlock ownership.

Note: Lifecycle wipes in `ALPHA` (Chapter 14) MAY delete cosmetics, badges, and/or GlobalStars regardless of normal yearly reset preservation rules.

Trigger + ordering promise (resource-facing):
- Timing authority is in Chapter 07.
- Persistence ordering/integrity records are in Chapter 13.
- The yearly reset triggers immediately after the system processes the **12th consecutive season expiration** of the current yearly cycle.
- The yearly reset executes as an **atomic global operation** during expiration finalization of that triggering expiration event.
- Ordering for the triggering season MUST be:
  1. apply seasonal bonuses,
  2. convert Seasonal Stars → Global Stars,
  3. record required pre-reset snapshots (Chapter 15) and integrity facts (Chapter 13),
  4. wipe Global Stars (cosmetics persist)

Season overlap/concurrency does not prevent yearly reset execution (season timing framework is defined in Chapter 07).

## 4.5 Sigils (Seasonal Strategic Resources)

### 4.5.1 Definition

Sigils are season-bound strategic resources used to activate temporary in-season effects.

Sigils:
- are not currency
- MUST NOT convert into Coins or Stars
- MUST be fungible within tier
- MUST have no inventory cap
- MUST exist only within a season
- MUST NOT persist beyond season exit (Lock-In or natural expiration only)

A player MAY hold unlimited Sigils if supply allows.

### 4.5.2 Tier Tokens (Authoritative)

Sigils are exactly five tier tokens (no sub-types):

- Tier I — Common
- Tier II — Uncommon
- Tier III — Rare
- Tier IV — Epic
- Tier V — Legendary

Tier gates which boosts and system events a player may activate. Effect mechanics are defined in **Chapter 03 (Boost Activation / System Events)**.

### 4.5.3 Acquisition Methods

Sigils are acquired through:
1. **Probabilistic drop distribution** (defined in this chapter)
2. **Purchase from the seasonal Sigil Vault** using Seasonal Stars (this chapter defines inventory and contention rules; pricing is in Chapter 05)

Coins MUST NOT purchase Sigils.

Vault inventory and drop distribution are independent systems.

### 4.5.4 Probabilistic Drop Distribution (Deterministic Replay)

#### 4.5.4.1 Phase Placement (Reference Only)

Sigil drop evaluation phase placement and tick suppressions are defined in **Chapter 03**.  
This chapter defines only the drop distribution rules, eligibility predicate inputs, and determinism requirements.

#### 4.5.4.2 Drop Eligibility (Start-of-Tick Snapshot)

Eligibility is evaluated **only** from the start-of-tick snapshot (Chapter 03).

Predicate:
`eligible(T) = online_snapshot(T) && participating_snapshot(T) && activity_state_snapshot(T) == Active`

Sigils MUST NOT drop for:
- Idle players
- Offline players
- Non-participating players

#### 4.5.4.3 Tick-Type Suppressions

Drop evaluation:
- MUST occur on normal ticks where allowed by Chapter 03.
- MAY occur during blackout.
- MUST NOT occur on:
  - the final valid economic tick, and
  - the expiration tick

Suppressions are enforced via Chapter 03 tick flags; this chapter does not define tick classification.

#### 4.5.4.4 Distribution Parameters (Authoritative)

#### Base per-tick drop parameters

- **DropRatePerTick** = **1/50000**  
  Interpretation: for each drop-eligible player on tick `T`, the server performs one Bernoulli trial with success probability `1/50000`, subject to throttles and pity rules below. Deterministic integer sampling is defined in Chapter 15.
- **MaxDropsPerPlayerPerTick** = **1**  
  Interpretation: even if multiple mechanisms would award drops on the same tick, the player receives at most one Sigil drop on that tick.
- **TierOddsTable_fp** (conditional on a drop being awarded and not forced by pity):
  - Fixed-point scale: `1_000_000 = 1.0`
  - Tier weights MUST sum to exactly `1_000_000`.
  - Weights:
    - Tier I: `700_000`
    - Tier II: `200_000`
    - Tier III: `80_000`
    - Tier IV: `15_000`
    - Tier V: `5_000`
  - CDF rule: sample `u ∈ [0, 999_999]` and pick the first tier where `u < cdf_fp[tier]` (deterministic sampling contract is defined in Chapter 15).

If a rule is not explicitly defined in this section, it MUST NOT exist.

#### Pity rule (forced Tier I at threshold)

Maintain `eligible_ticks_since_last_drop` per player:

- Increment it by **1** on each tick where the player is **drop-eligible**.
- Reset it to **0** when:
  - a Sigil drop is awarded to that player, **or**
  - the player is **not** drop-eligible on a tick.

If `eligible_ticks_since_last_drop >= 120000` at the start-of-tick eligibility evaluation on tick `T`, then award **exactly one** Sigil drop on tick `T` (bypassing `DropRatePerTick`), with **Tier forced to Tier I**, then reset the counter to **0**.

#### Throttle rule (rolling 86,400-tick cap)

Maintain a rolling-window counter `drops_last_86400_ticks` per player (window length = **86,400** ticks).

- A player MAY receive **at most 3** Sigil drops within any rolling 86,400-tick window.
- For tick `T` eligibility evaluation, the window is defined as drop awards with `award_tick ∈ [T-86400, T-1]` (i.e., the prior 86,400 ticks).
- If the player is at the cap on tick `T` (window count ≥ 3), then on that tick they are treated as **drop-ineligible**:
  - No RNG attempt occurs.
  - Pity counter does not increment on that tick.
  - No drop may be awarded by pity on that tick.

#### 4.5.4.4.1 Counter initialization and window boundaries (Authoritative)

- All Sigil-drop counters and rolling-window trackers initialize to `0` at season creation (the tick where the season becomes active; see Chapter 07).
- Rolling windows clamp to season start:
  - Ticks before season start do not exist for this system (treated as absent, not zero-filled).
  - For early ticks, any window `[T-N, T-1]` is interpreted as `[season_start_tick, T-1]`.
- Any pity/throttle logic MUST treat “ticks before season start” as absent.

#### 4.5.4.5 Lock-In Interaction

If a player has queued Lock-In intent but has not confirmed it, they remain eligible for Sigil drops (subject to start-of-tick snapshot).

If Lock-In resolves during a tick:
- any drops that occurred earlier in that tick remain valid,
- no additional drop suppression logic exists beyond Chapter 03 phase suppressions.

#### 4.5.4.6 Randomness and Replay Integrity (Reference + Local Draw Indexing)

Sigil drops are probabilistic but MUST be deterministically reproducible.

**Authoritative randomness contract (owner = Chapter 15).**
- The deterministic hash-based RNG contract (domain separation, encodings, `tick_seed`, `draw_bytes(i)`, and integer sampling rules) is defined in **Chapter 15**.
- Chapter 04 MUST NOT restate that contract.

**Local draw-index assignment (owned by this section).**
For each eligible player and tick `T` where drop evaluation is not suppressed by throttle and pity does not force an award:
- Use `draw_bytes(0)` for the Bernoulli `1/50000` trial.
- If the Bernoulli succeeds, use `draw_bytes(1)` for the tier roll using `TierOddsTable_fp`.
- No other draw indices are used by this system.

If pity forces an award on tick `T`, no RNG draws occur and Tier is forced to Tier I.

**Telemetry / audit requirement (owner = Chapter 15).**
- A Sigil drop award MUST emit a dedicated telemetry/audit event (`sigil_drop_awarded`) as defined in Chapter 15.
- A “no drop” per-tick event is NOT required.

### 4.5.5 Seasonal Sigil Vault

The Sigil Vault:
- has fixed inventory at season start,
- is globally shared,
- is not replenished.

Inventory structure:
- tier counts only (Tier I–V).

Initial inventory quantities per tier MUST be defined here:
- Tier I count: **2000**
- Tier II count: **800**
- Tier III count: **250**
- Tier IV count: **60**
- Tier V count: **15**

#### Vault pricing (PER_TIER step tables by remaining inventory)

The seasonal Sigil Vault sells **exactly one (1)** Sigil unit per purchase action (`purchase_sigils_vault`) at a cost in **Seasonal Stars**.

For each tick `T`, define `VaultCostStars_surface[T][tier]` by evaluating against `vault_remaining_snapshot(T)[tier]`:

- Pick the **first** matching `(remaining_inclusive_min, cost_stars)` pair in the tier’s table where `vault_remaining_snapshot(T)[tier] >= remaining_inclusive_min`.
- If `vault_remaining_snapshot(T)[tier] == 0`, the tier is **not purchasable** and has **no surface price**.

Locked quote rule (purchase actions):
- For `purchase_sigils_vault` with `intake_tick = T`, the server MUST compute a locked quote using `VaultCostStars_surface[T][tier]` and reserve that exact Seasonal Stars amount at acceptance (Chapter 03).

Tier I pricing table (remaining ≥ min → cost):
- (1600, 2), (1200, 3), (800, 4), (400, 6), (200, 9), (1, 13)

Tier II pricing table:
- (640, 6), (480, 8), (320, 11), (160, 15), (80, 20), (1, 28)

Tier III pricing table:
- (200, 18), (150, 24), (100, 32), (50, 44), (25, 60), (1, 85)

Tier IV pricing table:
- (48, 60), (36, 80), (24, 110), (12, 150), (6, 210), (1, 300)

Tier V pricing table:
- (12, 220), (9, 300), (6, 420), (3, 600), (2, 850), (1, 1200)

**Deterministic contention + FIFO validation.**
- Vault purchases are season-scoped actions and MUST follow Chapter 03 acceptance ordering and `accept_seq` assignment.
- Acceptance MUST validate current vault stock and MUST reserve the purchased unit at acceptance time.
- If stock is exhausted at the time a request is reached in deterministic acceptance order, that request MUST be rejected.

If multiple players attempt to purchase the final Sigil unit(s) such that contention would exist, tie-break MUST be deterministic by:

1. `intake_tick` (ascending)  
2. `accept_seq` (ascending)  
3. `player_id` (ascending)

Vault exhaustion:
- permanently removes that tier from vault availability,
- MUST NOT affect probabilistic drop eligibility.

Sigils already in player inventory remain usable and tradable even if vault stock is exhausted.

### 4.5.6 Trading

Sigils MAY be included in trades under the canonical trade system rules (owner = Chapter 05) and season action legality (owner = Chapter 03).

Additional Sigil-specific constraints (binding):
- Sigils MAY be traded only:
  - between current season participants (same season),
  - within the same season only,
  - outside blackout,
  - prior to Lock-In,
  - prior to expiration.
- Sigils MUST NOT be traded:
  - during blackout,
  - after Lock-In resolves,
  - after expiration resolves,
  - across seasons.

Trade composition constraints, fee rules, reservation/escrow timing, conflict/ordering keys, timeout rules, and blackout/expiration invalidation behavior are defined in:
- Chapter 05 (trade model + constraints + valuation/fees + conflict replacement), and
- Chapter 02 (trade lock timing + timeout + blackout/expiration terminalization).

### 4.5.7 Destruction

Sigils are destroyed when:
- spent to activate boosts or system events (effects defined in Chapter 03),
- Lock-In resolves,
- season expiration resolves.

Spend semantics:
- Sigil consumption MUST reserve at acceptance and commit at resolution.
- No refunds exist once an accepted spend resolves; invalidated actions must release reservations (per Chapter 03 defensive rules).

Sigils MUST NEVER convert into any other resource.

## 4.6 Cosmetic / Meta Unlocks

### 4.6.1 Definition

Cosmetic unlocks are permanent, non-economic profile enhancements.

They are purchased exclusively using Global Stars.

Cosmetics are discrete SKUs with permanent ownership (owned / not owned). No alternative permanent currency exists.

### 4.6.2 Constraints

Cosmetics:
- MUST NOT alter UBI rate
- MUST NOT alter Sigil drop rate
- MUST NOT alter vault inventory or access
- MUST NOT alter participation timers
- MUST NOT alter leaderboard ordering
- MUST NOT alter economic resolution
- MUST NOT provide mechanical advantage

Cosmetics are purely representational.

### 4.6.3 Visibility

Cosmetics:
- MUST be visible on player profiles (to others and spectators),
- MUST NOT be shown on any leaderboard surface.

---

## 4.7 Resource Invariants

The following invariants apply to all resources:

1. No resource may be created outside its defined creation path.  
2. No resource may be manually granted.  
3. No resource may be removed by admin action.  
4. No resource may become negative.  
5. No resource may bypass blackout, Lock-In, expiration, or yearly reset constraints.  
6. Only Global Stars and Cosmetics persist beyond seasonal boundaries.  
7. No resource is artificially capped (bounded only by int64-domain validation).  
8. UI magnitude shorthand is visual-only and never used in calculations or ordering.  
9. No resource may create permanent mechanical advantage across seasons.  

Cosmetics are naturally capped by the finite SKU catalog size; this is a catalog-size cap, not an economic ceiling.
