# CHAPTER 11 — EDGE CASES AND FAILURE STATES

This chapter defines failure states and recovery rules for client/server behavior.
“Failure states” here refer to technical and state-recovery failures, not player outcome framing (see Chapter 09).

**Authority split (binding):**
- Chapter 10 owns modal/UI behavior and clickability rules.
- Chapter 03 owns season-scoped action legality, intake buffering, ordering, and tick-phase execution.
- Chapter 15 owns the closed `reason_code` catalog and audit/telemetry schema.
- Chapter 07 owns time boundaries (blackout, expiration tick).
- Chapter 13 owns persistence ordering, idempotency uniqueness, and retention.

---

## 11.1 Disconnects and Reconnects

Disconnects are a client connectivity condition.
Disconnect UI is a modal (Chapter 10) and blocks interaction behind it.

The server MUST treat disconnect as transport loss; it MUST NOT infer player intent from disconnect.

**Snapshot boundary (reference):**
- `online_snapshot(T)` is sampled at `season_tick_start(this_season, T)` and does not change mid-tick (Chapter 03).

---

## 11.2 Failure Taxonomy (Reason Codes + Failure Classes)

### 11.2.1 Reason codes (binding)

All rejected actions MUST return a `reason_code` from Chapter 15’s closed set.

This chapter MUST NOT introduce new `reason_code` identifiers.

### 11.2.2 Failure classes (UI grouping; non-authoritative)

Failure classes are a UI/UX taxonomy for grouping failures and gating.

Minimum failure classes referenced by this Bible:

- `IDLE_ENFORCED`
- `DISCONNECTED`
- `MODE_DISALLOWS_ACTION`
- `NOT_ELIGIBLE`
- `ILLEGAL_IN_BLACKOUT`

FailureClass strings are non-authoritative labels; server authority is carried by `reason_code` and gating flags.

---

### 11.2.3 ServerMode (authoritative)

`ServerMode` is a single global server-authoritative value used to gate connections and/or actions during operations and incident response.

Closed set (binding):
`ServerMode ∈ {NORMAL, MAINTENANCE_LOCKDOWN, READ_ONLY_ECONOMY, LOCKDOWN_CONNECTIONS, RATE_LIMIT_ACTIONS}`

Per-mode gating matrix (binding):

* **NORMAL**
  * allow_connections: Yes
  * allow_login: Yes
  * allow_season_actions_non_exempt: Yes
  * allow_global_actions: Yes
  * allow_chat_send: Yes
  * allow_reads/snapshots: Yes

* **MAINTENANCE_LOCKDOWN**
  * allow_connections: No (`reason_code=lockdown_connections`)
  * allow_login: No (`reason_code=lockdown_connections`)
  * allow_season_actions_non_exempt: No (`reason_code=mode_disallows_action`)
  * allow_global_actions: No (`reason_code=mode_disallows_action`)
  * allow_chat_send: No (`reason_code=mode_disallows_action`)
  * allow_reads/snapshots: Yes

* **READ_ONLY_ECONOMY**
  * allow_connections: Yes
  * allow_login: Yes
  * allow_season_actions_non_exempt: No (`reason_code=read_only_economy`)
  * allow_global_actions: No (`reason_code=read_only_economy`)
  * allow_chat_send: Yes
  * allow_reads/snapshots: Yes

* **LOCKDOWN_CONNECTIONS**
  * allow_connections: No (`reason_code=lockdown_connections`)
  * allow_login: No (`reason_code=lockdown_connections`)
  * allow_season_actions_non_exempt: Yes (existing sessions only)
  * allow_global_actions: Yes (existing sessions only)
  * allow_chat_send: Yes (existing sessions only)
  * allow_reads/snapshots: Yes

* **RATE_LIMIT_ACTIONS**
  * allow_connections: Yes
  * allow_login: Yes
  * allow_season_actions_non_exempt: Yes (may reject bursts with `reason_code=rate_limited`)
  * allow_global_actions: Yes (may reject bursts with `reason_code=rate_limited`)
  * allow_chat_send: Yes (may reject bursts with `reason_code=rate_limited`)
  * allow_reads/snapshots: Yes

Mode transition semantics (binding):
- Any ServerMode change MUST become effective at the next tick boundary.
- Actions already accepted before the mode becomes effective MUST settle under their already-accepted legality snapshot.
- New accepts after the effective tick MUST be evaluated under the new mode and rejected when disallowed, using the reason_code listed above.

## 11.3 Idle Handling (Backend Contracts)

Idle is a server-enforced activity state that blocks season-scoped economy actions until acknowledgement.

Idle state is represented as:
- server flag: `idle_modal_active == true`
- FailureClass: `IDLE_ENFORCED`
- fields: `idle_state`, `idle_since_tick` (**global_tick_index**), `idle_cause ∈ {INACTIVITY, ABUSE_COOLDOWN, POLICY_LOCK}`

### 11.3.1 Inactivity-triggered idle (authoritative)

**Trigger threshold (hard rule):** the server MUST enforce inactivity-idle after **900 seconds** (fifteen (15) minutes) with **no qualifying Activity Decision** while the player is economically participating in a season.

Determinism contract:

- The inactivity window is measured in **committed ticks**, using `global_tick_index` as the time base.
- The server MUST track `last_activity_decision_global_tick_index` per `(player_id, season_id)` (implementation detail; name MAY vary).
  - It MUST be updated only when a **qualifying Activity Decision** commits.
  - The qualifying Activity Decision set is defined by Chapter 03 (counts-for-activity inputs).
- On each tick commit `G` while `joined_season_id == S` and `participation_enabled == true`:
  - If `G - last_activity_decision_global_tick_index >= 900`, the server MUST set `FailureClass=IDLE_ENFORCED` with `idle_cause=INACTIVITY`.
  - `idle_since_tick` MUST equal the `global_tick_index` `G` on which the idle state became enforced.
- Re-enforcement while already idle is an idempotent no-op (do not move `idle_since_tick`).

Notes:

- Disconnects do not “pause” inactivity. If a player is disconnected and accrues no qualifying Activity Decision for ≥ 900 seconds, they MAY become idle; UI precedence remains owned by Chapter 10 (Disconnected overrides Idle label).
- Idle enforcement MUST NOT stop `participation_time_total` accumulation; it only gates season-scoped inputs and may affect economy rates per Chapter 05.

### 11.3.2 Idle gating scope

While idle is enforced, the server MUST reject any season-scoped non-exempt economy action with:

- FailureClass: `IDLE_ENFORCED`
- `reason_code = idle_gated` (Chapter 15)

Exempt actions while idle:
- `idle_modal_ack`
- chat (global + DM + season chat eligibility rules still apply; idle does not block chat)

### 11.3.3 Idle persistence across context

Idle enforcement persists across:
- viewing/spectating other seasons
- global context surfaces
- reconnects

Idle is a server state; client MUST display it only when server flag indicates.

### 11.3.4 Idle vs pending accepted action

If an action was accepted prior to idle enforcement and is pending resolution, idle enforcement MUST NOT retroactively cancel acceptance.
Resolution MUST proceed per the action’s rules unless that action is explicitly invalidated by mode/eligibility at resolution.

### 11.3.5 Clearing idle

Idle is cleared only by:
- `idle_modal_ack` committing (Chapter 03)
- explicit server-side policy/system unlock (for non-inactivity causes)

Client MUST NOT clear idle locally.
Chat activity MUST NOT clear idle.

## 11.4 Season End Edge Cases

### 11.4.1 Expiration tick boundary rejections (binding)

Time boundaries are owned by Chapter 07. Action acceptance/rejection is owned by Chapter 03.

Minimum external behavior (binding):

- Non-exempt season-scoped gameplay actions that would resolve on expiration tick `E` MUST be rejected at acceptance.
- Any non-exempt season-scoped gameplay action whose `intake_tick == E` MUST be rejected with `reason_code = expiration_tick` (Chapter 15).

### 11.4.2 Expiration finalization retries (binding)

Expiration finalization MUST be safe to retry and MUST NOT double-apply:
- bonuses,
- conversions,
- badge awards,
- trade escrow releases,
- destructive cleanup.

Idempotency and checkpoint requirements are owned by Chapter 13.

### 11.4.3 Season end acknowledgements (binding)

`season_end_ack(target_season_id)` is post-expiration and wall-clock (Chapter 03).

Retention behavior:
- `season_end_ack` records MAY be compacted after a retention window (Chapter 13).
- After compaction, a late `season_end_ack` MUST be a deterministic no-op success (idempotent success; no error).

---

## 11.5 Lock-In Edge Conditions

### 11.5.1 Blackout illegality (binding)

Lock-In is illegal during blackout (Chapter 07) and MUST be rejected when attempted during blackout.

Rejection MUST use a Chapter 15 `reason_code` that indicates blackout illegality:
- `blackout_disallows_action`.

### 11.5.2 Minimum participation gate (binding)

The minimum participation gate predicate is defined in Chapter 03.

If `lock_in_confirm` is rejected due to the Chapter 03 minimum participation gate, the rejection MUST use:

- `reason_code = not_eligible`.


### 11.5.3 Multiple Lock-Ins (binding)

Multiple successful Lock-Ins in the same season are allowed (Chapter 03 / Chapter 07).

Failure behavior:
- Lock-In attempts that are rejected MUST NOT affect stored Lock-In history, balances, or participation state.
- Successful Lock-Ins MUST be idempotent via the Chapter 13 uniqueness key `(player_id, season_id, lock_in_effect_tick)`.

---

## 11.6 UI and State Recovery Rules

### 11.6.1 Snapshot refresh after rejection (binding)

On any server rejection:
- Client MUST display a toast or inline notice derived from `reason_code`.
- Client MUST request a fresh authoritative snapshot before enabling further economy actions.

### 11.6.2 Duplicate requests and idempotency (binding)

If the client resubmits an identical request due to uncertainty, the server MUST enforce idempotency per Chapter 15/Chapter 13 contracts:

- If the request is a true idempotent duplicate: server MUST return an idempotent success response (or “already applied”) and MUST NOT double-apply side effects.
- If the request reuses an idempotency identity incorrectly: server MUST reject with `reason_code = client_request_id_reused` (Chapter 15).
